# ScreenTinker Server-Side Audit — 2026-05-28

**Branch:** `screen-share`  **HEAD:** `457d7d4`  **Auditor:** general-purpose subagent  **Scope:** `server/**` only

## P0 (production-blocker, fix immediately)

### Finding: Public `/api/content/:id/file` endpoint allows unauthenticated downloads of ANY content
**File:** `server/server.js:323-339`
**Issue:** This endpoint is mounted BEFORE `requireAuth` (devices need to fetch without JWT), but the only gate is "is this content referenced by some playlist_item OR widget config anywhere in the system?" — no workspace check. Anyone who learns a content UUID can fetch the file. The `inWidget` LIKE scan runs over ALL widgets cross-tenant.
**Reproduction:** `curl https://media.mbfdhub.com/api/content/<any-uuid-currently-in-a-playlist>/file` — returns the file with no auth.
**Fix:** Require either a device_token (matching a device whose playlist contains the content) OR a JWT with workspace access. At minimum: scope the playlist/widget lookup to widgets/playlists in the SAME workspace as the content, and require a short-lived signed URL (HMAC) for device fetches.

### Finding: Per-playlist-item fit/fullscreen toggle does not exist anywhere in the schema or API
**File:** `server/db/schema.sql:345-355`, `server/routes/playlists.js`, `server/routes/assignments.js`, `server/ws/deviceSocket.js:68-149`
**Issue:** `playlist_items` has no `is_fullscreen`, `fit_mode`, or `display_mode` column. The only `fit_mode` lives on `layout_zones` (line 144). Player at `player/index.html:1561,1567` always uses `zone.fit_mode || 'cover'`. No per-item override. Known P1 #1 confirmed.
**Reproduction:** Add a portrait video to a landscape playlist — it gets cropped (cover) or letterboxed (contain) for the WHOLE playlist, no per-item knob.
**Fix:**
1. `ALTER TABLE playlist_items ADD COLUMN fit_mode TEXT` (null=inherit zone, 'cover', 'contain', 'fill').
2. Plumb through `buildSnapshotItems()` in routes/playlists.js so the field lands in `published_snapshot`.
3. Accept `fit_mode` in `POST /:id/items` and `PUT /:id/items/:itemId`.
4. Update `routes/assignments.js` POST and PUT handlers.

### Finding: No YouTube transcoding — YouTube content stays as iframe embeds that fail on multi-tile walls
**File:** `server/routes/content.js:202-240`
**Issue:** `POST /youtube` stores `mime_type='video/youtube'`, `remote_url=<embed URL>`. Player renders as iframes (`index.html:1525`). Iframes can't be CSS-transformed across multiple displays on a 3-tile wall without GPU texture overflow >4096px (commit `dbf92ce` revert documents this). Desired fix: server-side transcode via yt-dlp at upload. None of the codebase invokes yt-dlp/youtube-dl. Known P1 #2 confirmed.
**Reproduction:** Add a YouTube link → assign to a 3-tile wall → only the middle tile plays.
**Fix:**
1. Add `yt-dlp` as system dep; fork from `POST /youtube` into a background download into `config.contentDir`.
2. Store result as `mime_type='video/mp4'`, `filepath=<uuid>.mp4`, `remote_url=null`.
3. Use `execFile('yt-dlp', ['-f', 'best[height<=1080]', '-o', dest, videoUrl], { timeout: 600_000 })`.
4. Show "processing" state to user (insert row with `mime_type='video/youtube-processing'`, finalize when done).
5. Best-format selection should cap height ≤ 1080 to stay within Fire TV decode limits.

### Finding: Per-device manual display-resolution override is impossible via the API
**File:** `server/routes/devices.js:142-164`, `server/routes/video-walls.js:217-285`
**Issue:** `routes/devices.js` PUT whitelists ONLY `['name', 'notes', 'timezone', 'orientation', 'default_content_id']` — `screen_width`/`screen_height` cannot be written by an admin. The only writer is `ws/deviceSocket.js:340-341,418-419` from `device:register`, which trusts whatever `device_info` the player sends. There is no `refresh_rate_hz`, `canvas_width`, or `canvas_height` override on `devices`. Known P1 #4 confirmed.
**Fix:**
1. Add migrations: `screen_width`, `screen_height` (already exist on `devices`) plus `refresh_rate_hz INTEGER`, `auto_detect_resolution INTEGER NOT NULL DEFAULT 1`.
2. Extend `ALLOWED_FIELDS` in `routes/devices.js:148` to include these.
3. In `ws/deviceSocket.js:340-341` and 418-419, ONLY overwrite `screen_width`/`screen_height` when `auto_detect_resolution=1`. Admin override is canonical otherwise.

### Finding: No PDF/PowerPoint/Word/Excel upload support; ONLYOFFICE not wired
**File:** `server/middleware/upload.js:31-42`, `server/routes/content.js:88-173`
**Issue:** `fileFilter` whitelists only `video/*` and `image/*` MIME prefixes. PDF/Office docs rejected at multer. ONLYOFFICE at `office.mbfdhub.com` is unused. Known issue #5 confirmed.
**Fix:**
1. Extend whitelist for `application/pdf` + Office MIME types.
2. After multer write, fork a background ONLYOFFICE conversion to PDF (POST to `office.mbfdhub.com`'s ConversionService).
3. For thumbnails: use `pdftoppm` on first page when MIME is `application/pdf`.

## P1 (significant bug, fix this cycle)

### Finding: Layout duplicate orphans playlist_item.zone_id references — confirmed root cause of "templates broken"
**File:** `server/routes/layouts.js:209-217`
**Issue:** When duplicating a layout (or applying a template), new UUID zone IDs are minted. Any `playlist_items.zone_id` referring to the source layout's zones is now orphaned. The player at `player/index.html:1540` does `playlist.find(a => a.zone_id === zone.id)` and falls through to `defaultItem` (current playlist item) → same content in every zone, looking like a layout bug.
**Fix:** Give every layout-zone an immutable `slot_key` (`'main'`, `'side'`, `'ticker'`) and resolve `playlist_items.zone_id` against `slot_key` at payload-build time. OR preserve zone IDs on duplicate.

### Finding: Express has no error-handling middleware; default handler leaks stack traces in 500s
**File:** `server/server.js` (no `app.use((err, req, res, next) => …)` anywhere)
**Fix:** Add at bottom of `server.js` before `app.get('*')`:
```js
app.use((err, req, res, next) => {
  console.error('[unhandled]', err.stack || err.message);
  if (err instanceof multer.MulterError) {
    return res.status(400).json({ error: `Upload error: ${err.code}` });
  }
  res.status(err.status || 500).json({ error: 'Internal server error' });
});
```

### Finding: SSRF allow-list for `remote_url` misses private-network patterns
**File:** `server/routes/content.js:35-51`
**Issue:** Misses DNS rebinding, IPv4 alt encodings (decimal int, octal, 0x), `::ffff:127.0.0.1`, Tailscale 100.64/10.
**Fix:** Resolve via `dns.lookup` then check resolved IP against private ranges. Pin to resolved IP. Add Tailscale CGNAT range.

### Finding: No automated SQLite backup
**File:** `server/db/database.js`
**Fix:** Add `services/db-backup.js` invoked from `server.js` doing `db.backup(dest)` every 6h with 30-day rotation. Push to `/mnt/mbfd-storage/backups/`.

## Notes & Observations

- **Workspace gating is consistent** across nearly every CRUD route. Phase 2.2 closed many cross-tenant leaks; `lib/tenancy.js:accessContext()` is invoked uniformly.
- **SQL injection: no findings.** Every query in scope uses parameterized statements.
- **Rate limiting** exists for auth/pairing/content/status export, but not for `/api/walls`, `/api/devices` mutations.
- **CSP looks reasonable** for dashboard pages — strict `scriptSrc 'self'`, no `'unsafe-inline'` on scripts. `frameSrc` allows YouTube. `frameAncestors` allows `cloud.mbfdhub.com`.
- **No secrets found in source.** All credentials come from env vars.
- **Cloudflare proxy trust correctly scoped** to published CF IP ranges.
- **Screen-share signaling is well-isolated** with per-socket ICE rate limiting, broadcaster preemption, reaper for stale sessions, consistent `canActOnDevice` gates.
- **Fingerprint-reclaim guard in `deviceSocket.js:240-298`** is hardened with clear comments. Good.
- **`maxFileSize: 2GB`** intentional; Cloudflare's edge will reject earlier on most plans — document for operators.
- **Activity log auto-logger** sometimes captures wrong ID (e.g., for `PUT /workspaces/:id/members/:userId` captures workspace as device_id). Cosmetic.
