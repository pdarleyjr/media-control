## P0 (production-blocker)

### Finding: Fingerprint-reclaim "fall through" (457d7d4) is server-only; player still wipes credentials on `device:auth-error`
**File:** `server/player/index.html:704-714`
**Issue:** Commit 457d7d4 only edits `server/ws/deviceSocket.js`. Player's `device:auth-error` handler still deletes `deviceId`/`deviceToken`, sets `paired=false`, shows "Authentication failed" with **pairing section hidden** (`pairingSection.style.display = 'none'`). The new pairing code stored in `config.pairingCode` (line 864) is never surfaced.
**Reproduction:** Trigger `auth-error` branch on server (unknown device_id + invalid device_token). Player shows error text only, no recovery short of pressing Escape (impossible on a Fire TV with no keyboard).
**Fix:** In `device:auth-error` handler, if `config.pairingCode` exists, show `#pairingSection` with that code instead of just the error text. Stop hiding `#pairingSection` unconditionally.

### Finding: Wall follower `tapOverlay` cold-load races screen-share unmute → follower audio leak
**File:** `server/player/index.html:540-564, 614`
**Issue:** Cold-load `tapOverlay.onclick` fires BEFORE `applyWallMode()` runs — `wallConfig` is null, `isWallFollower()` returns false, audio gets unlocked, the next playlist payload promoting this device to follower-mode inherits unmuted videos.
**Fix:** Guard `unlockAudio()` against the boot-time tap as well; or apply wall config BEFORE allowing tap-to-unlock.

### Finding: Wall follower stuck at start-of-item until next 250ms sync after content swap
**File:** `server/player/index.html:716-730, 1130-1158`
**Issue:** When leader advances via `wall:sync`, follower runs `playCurrentItem()` → new `<video>` at `currentTime=0`. Followers wait for next periodic 250ms sync tick. Worse: order is `renderContent()` then `emitWallSync()` (lines 1187, 1191), so leader broadcasts position before its own renderContent finishes loading.
**Fix:** When follower receives `wall:sync` with changed `current_index`, after `playCurrentItem()` returns, immediately apply `data.position_sec + latency` as new video's `currentTime` (wait for `loadedmetadata`). Don't rely on next periodic tick.

### Finding: YouTube on wall followers — each tile loads own iframe with no slice mechanism (KNOWN P1 #2)
**File:** `server/player/index.html:1239-1382, 1442-1454`
**Issue:** Confirmed via code reading + repo history. In wall mode, YT iframe is mounted inside `wall-stage` positioned outside device viewport for non-center tiles. `#playerContainer.wall-mode` has `overflow: hidden`, iframe forced to `position: static; width: 100%; height: 100%` of stage → stage is 300vw wide for 3-tile wall, iframe is 300vw wide. YouTube letterboxes inside its own iframe; the actual video pixels occupy only a centered portion → **only middle tile's 100vw window intersects the video pixels**.
**Fix:** Per the brief, approach (a) — server-side transcode via yt-dlp to MP4. Player's HTML5 video path works in wall mode (line 1455-1507 uses stage mounting + `object-fit:fill`). No player-side fix needed once server delivers MP4.

## P1

### Finding: Solo-mode video uses `object-fit: contain` with no per-item override (KNOWN P1 #1)
**File:** `server/player/index.html:1473-1475`
**Issue:** Hardcoded ternary `wallConfig ? 'object-fit:fill' : 'object-fit:contain'`. No flag on playlist item to override. Schema (per `handlePlaylistUpdate` line 1071) has no `fit_mode`/`object_fit` field.
**Fix:**
```js
video.style.cssText = wallConfig
  ? `width:100%;height:100%;object-fit:fill;background:#000`
  : `width:100%;height:100%;object-fit:${item.object_fit || 'contain'};background:#000`;
```
Apply same to `isImage` branch at line 1511-1513.

### Finding: Zones/templates code path bypassed when `wallConfig` is set
**File:** `server/player/index.html:1449-1451`
**Issue:** `if (layout && layout.zones && layout.zones.length > 1 && !wallConfig) renderZones(...)`. Zones cannot combine with wall mode — silently ignored. If `playlist` items have no `zone_id`, every zone falls through to `defaultItem` → same content in every zone, looking like a layout bug. **Cross-confirmed with server audit's "Layout duplicate orphans zone_id" finding.**
**Fix:** Trace server's playlist-update emit to confirm `zone_id` is set. Add player-side warning log when `layout.zones.length > 1` and no assignment has `zone_id`. Document wall-excludes-zones limitation or implement zone-in-wall.

### Finding: `renderZones()` has no follower-aware advance logic
**File:** `server/player/index.html:1532-1574`
**Issue:** If zone-in-wall ever enabled, followers would self-advance and break sync.
**Fix:** Mirror `isFollower` checks from `renderContent()` (lines 1477, 1480, 1516, 1520, 1527) into `renderZones()`.

### Finding: Zone videos never registered as `currentVideoEl`, so wall-sync drift correction can't reach them
**File:** `server/player/index.html:1554-1563`
**Fix:** Assign zone video to `currentVideoEl` for the active zone; mirror teardown.

### Finding: `tapOverlay` cold-load path can race with auto-dismiss `connect()` call
**File:** `server/player/index.html:540-564`
**Fix:** Use `tapOverlayConsumed` flag set true in both branches; both call sites short-circuit if set.

### Finding: Service worker caches `/player/sw.js` itself
**File:** `server/player/sw.js:29`
**Fix:** Exclude `sw.js`: `if (url.pathname.startsWith('/player') && url.pathname !== '/player/sw.js')`.

### Finding: `screen-share-receiver` reads `window.socket` as fallback — no such global
**File:** `server/player/screen-share-receiver.js:52`
**Fix:** Explicitly expose `window.__playerState = { socket }` after the `io()` call at line 644.

### Finding: Pairing-code dead-end on server restart (config has deviceId+token but DB row reset)
**File:** `server/player/index.html:524-590`
**Issue:** Compounds with P0 #1. User sees only "Authentication failed" — no pairing code on screen.
**Fix:** Same as P0 #1 — show pairing code on auth-error if server included one, or generate locally and re-register immediately.

## Notes

- **Wall-sync reconnect path** is solidly covered: follower emits `wall:sync-request`, leader handles and emits immediate sync, `playCurrentItem` also pushes sync. Reconnect aligns within ~latency + 250ms. Weakness is content-swap timing (P0 above).
- **`reconnection: Infinity`** with 10s max delay — good for kiosks.
- **`wakeLock`** is requested every visibilitychange — good.
- **`tryUnmuteLeader()`** does pause+play re-prime — harmless on Silk.
- **No screenshot rate-limit** — `device:remote-start` triggers `setInterval(captureAndSend, 1000)` until `remote-stop`. Add max duration / heartbeat check.
- **SW filename version**: bumping `CACHE_NAME` from `v9` to `v10` is the only way to bust caches. Document in deploy runbook.
- **Fingerprint** is deterministic on identical hardware (the very reason 457d7d4 was needed). Consider including a `crypto.randomUUID()` persisted in localStorage with canvas hash as secondary signal.
