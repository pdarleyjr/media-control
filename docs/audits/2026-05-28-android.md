# ScreenTinker Android Native Player Audit ŌĆö 2026-05-28

**Branch:** `screen-share`  **HEAD:** `457d7d4`  **Auditor:** kotlin-reviewer subagent
**Scope:** `android/**` only

**Verdict: BLOCK** ŌĆö P0 infinite-update-loop (no version comparison), unbounded WakeLock, mutable PendingIntent for APK install, unrestricted ADB backup of credentials are all production blockers.

## P0 (Critical/Security)

### WakeLock acquired without timeout ŌĆö permanent CPU lock on device fleet
**File:** `android/app/src/main/java/com/remotedisplay/player/service/WebSocketService.kt:68-69`
**Fix:** Use timed acquire `wakeLock?.acquire(30 * 60 * 1000L)` re-acquired on heartbeat. Call `setReferenceCounted(false)` for safety.

### UpdateChecker blindly trusts server `update_available` flag ŌĆö no version comparison performed
**File:** `android/app/src/main/java/com/remotedisplay/player/service/UpdateChecker.kt:72-82`
**Issue:** The infinite-update-loop bug. Client reads `update_available` boolean and `latest_version` string but **performs zero version comparison**. If server returns `update_available: true` with `latest_version` Ōēż installed, device re-installs same APK every 30 minutes indefinitely. Also no WiFi check before ~50 MB APK download.
**Fix:**
```kotlin
if (updateAvailable && downloadUrl.isNotEmpty() && isNewerVersion(latestVersion, currentVersion)) {
    downloadAndInstall(...)
}

private fun isNewerVersion(latest: String, current: String): Boolean {
    return try {
        val l = latest.split(".").map { it.toInt() }
        val c = current.split(".").map { it.toInt() }
        (0..2).forEach { i ->
            val li = l.getOrElse(i) { 0 }; val ci = c.getOrElse(i) { 0 }
            if (li > ci) return true
            if (li < ci) return false
        }
        false
    } catch (_: Exception) { false }
}
```
Plus: `val wm = context.getSystemService(Context.WIFI_SERVICE) as WifiManager; if (!wm.isWifiEnabled) return`.

### `PendingIntent.FLAG_MUTABLE` for APK install callback ŌĆö intent hijacking risk
**File:** `android/app/src/main/java/com/remotedisplay/player/service/UpdateChecker.kt:172`
**Fix:** `if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.S) FLAG_IMMUTABLE else FLAG_MUTABLE`.

### `android:allowBackup="true"` with no backup rules ŌĆö device token leak via ADB
**File:** `android/app/src/main/AndroidManifest.xml:20`
**Issue:** ADB backup extracts `remote_display_secure` EncryptedSharedPreferences file, regular `remote_display` prefs (pairing code, device ID), and content cache. The KeyStore key is device-bound but encrypted bytes plus metadata are exfiltrated; the **plaintext fallback prefs** (see P2 finding) are fully readable. Meaningful data exposure for public-sector deployment.
**Fix:** `android:allowBackup="false"` OR add `android:fullBackupContent="@xml/backup_rules"` with explicit exclusions.

## P1 (High ŌĆö Must Fix Before Merge)

### DeviceInfo reports surface resolution, not physical display ŌĆö root cause of underreporting (KNOWN P1 #4)
**File:** `android/app/src/main/java/com/remotedisplay/player/telemetry/DeviceInfo.kt:126-131`
**Issue:** Calls `wm.defaultDisplay.getRealMetrics(dm)` returning `widthPixels`/`heightPixels`. On Fire TV 4K and Cube, OS renders at 1080p internally and upscales to 4K at compositor. `getRealMetrics()` returns surface res (1920├Ś1080), not physical panel. Server receives 1920├Ś1080 when wall is 12372├Ś2160.
**Fix (two-part):**
1. **Device telemetry:** use `Display.getSupportedModes()` to find highest-refresh physical mode:
```kotlin
val display = wm.defaultDisplay
val mode = display.supportedModes.maxByOrNull { it.physicalWidth.toLong() * it.physicalHeight }
if (mode != null) {
    dm.widthPixels = mode.physicalWidth
    dm.heightPixels = mode.physicalHeight
    return dm
}
```
2. **Admin override path:** add `canvas_width`/`canvas_height` to `ServerConfig` (EncryptedSharedPreferences). Server includes target canvas in payload. `getDeviceInfo()` prefers server-dictated canvas over device-reported. **This is the correct architectural fix for a mosaic wall** ŌĆö no single device knows the total canvas size.

### `ScreenShareReceiver` no bitrate cap on answer
**File:** `android/app/src/main/java/com/remotedisplay/player/service/ScreenShareReceiver.kt:291-321`
**Fix:** After `setLocalDescription` succeeds, walk receivers, set `encodings[0].maxBitrateBps = 2_500_000`. Or SDP munge `b=AS:2500`.

### `ScreenShareReceiver` no reconnect path after ICE FAILED/DISCONNECTED
**File:** `android/app/src/main/java/com/remotedisplay/player/service/ScreenShareReceiver.kt:193-216`
**Fix:** On teardown from ICE failure, emit `device:screen-share-reconnect-request` with `reason: "ice_failed"`. Server signaling re-triggers offer after delay. Or implement ICE restart via `peerConnection?.restartIce()`.

### `ContentCache` no size limit or LRU eviction ŌĆö disk fills silently
**File:** `android/app/src/main/java/com/remotedisplay/player/data/ContentCache.kt:1-68`
**Fix:** `MAX_CACHE_BYTES` constant (~4 GB), LRU eviction before download, partial-file cleanup in catch.

### `ScreenCapturePermissionActivity` stores MediaProjection result in static companion ŌĆö memory leak + security risk
**File:** `android/app/src/main/java/com/remotedisplay/player/ScreenCapturePermissionActivity.kt:26-34`
**Fix:** Remove static `resultData`/`resultCode`/`hasPermission`. Pass result directly to `ScreenCaptureService.startProjection()`.

### `isMinifyEnabled = false` for release ŌĆö no code shrinking, no obfuscation
**File:** `android/app/build.gradle.kts:32`
**Fix:** Set `isMinifyEnabled = true`. Expand `proguard-rules.pro` to keep WebRTC, Gson, ServerConfig classes.

## P2 (Medium)

- **Unbounded `Thread().start()` throughout** (WebSocketService, UpdateChecker, MediaPlayerManager, ZoneManager, TouchInjector, PowerAccessibilityService). No pool, no lifecycle. Screenshot stream stacks threads under load. Replace with coroutine scope tied to service lifecycle.
- **`DeviceInfo.getCpuUsage()`** returns JVM heap ratio, not CPU usage. Read `/proc/stat` or rename field.
- **`ZoneManager.renderAssignments()`** fires raw Thread for remote images but never cancels on cleanup ŌåÆ bitmap leak per update cycle.
- **`PowerAccessibilityService` static `instance`** field ŌĆö accepted pattern for accessibility services; document as such and ensure callers null-check (they do).
- **`usesCleartextTraffic="true"` globally** with no `network_security_config.xml` ŌĆö all traffic to provisioned server URL could be HTTP.
- **`MainActivity` content download** uses `thread { ... }` then `handler.post` to UI ŌĆö no lifecycle guard. Guard with `if (!isDestroyed && !isFinishing)` or migrate to ViewModelScope.
- **`ServerConfig` fallback** silently stores credentials in plaintext when EncryptedSharedPreferences fails. Should clear prior creds, emit server-side telemetry, prompt re-provision (or fail hard).

## P3 (Low)

### No `wall:sync` protocol implementation ŌĆö native player drifts on multi-display walls
**Files:** All player files
**Issue:** Web player implements `wall:sync` leader/follower. Native player advances independently per-device based on ExoPlayer `STATE_ENDED`. No `wall:sync` Socket.IO listener. **Functional gap** for synchronized walls.
**Fix:** Implement `wall:sync` subscriber in `WebSocketService.kt`; MainActivity then seeks ExoPlayer to target position and jumps PlaylistController to target index.

### No `is_fullscreen` per-item flag in `PlaylistItem`
**File:** `android/app/src/main/java/com/remotedisplay/player/player/PlaylistController.kt:9-23`
**Fix:** Add `val isFullscreen: Boolean = false`, parse from `obj.optInt("is_fullscreen", 0) == 1`. In MediaPlayerManager set `playerView.resizeMode = if (isFullscreen) RESIZE_MODE_ZOOM else RESIZE_MODE_FIT`.

### Other P3
- `accessibility_service_config.xml` requests `typeAllMask` + `canRetrieveWindowContent` ŌĆö excessive. Use `typeWindowStateChanged`, drop content retrieval.
- `ScreenshotCapture.captureOnMainThread()` blocks main thread up to 3 seconds via CountDownLatch. Use `PixelCopy` (API 26+) for off-main capture.
- `UpdateChecker` no SHA-256 verification of downloaded APK before install. Server should include `sha256` in update response.
- `BootReceiver.onReceive()` does blocking work inline ŌĆö ANR risk. Use `goAsync()`.
- Manifest declares `READ/WRITE_EXTERNAL_STORAGE` but never requests at runtime. Remove (not needed for `getExternalFilesDir()`).

## Notes

- **Wall sync gap is a feature gap, not a regression.** No evidence the native player ever implemented `wall:sync`. Needs server-side protocol spec.
- **WebRTC bitrate caps.** Receiver-side `RtpParameters` is best-effort (controls RTCP REMB hint). More reliable: cap sender-side in `server/ws/screen-share-signaling.js` before offer is constructed.
- **Coroutine migration path.** Project has `kotlinx-coroutines-android:1.7.3` in deps but zero usage ŌĆö dead weight. Migration eliminates entire class of P2 threading issues.
- **`EncryptedSharedPreferences` alpha dependency.** `androidx.security:security-crypto:1.1.0-alpha06` is alpha. Stable is `1.0.0`. Downgrade unless alpha features specifically required.
